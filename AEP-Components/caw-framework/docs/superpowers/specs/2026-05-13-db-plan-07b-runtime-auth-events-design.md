# DB Plan 07b Runtime Bundle, Listener Auth, And Bypass Events Design

Date: 2026-05-13
Status: Approved design for implementation planning

## Goal

Plan 07b connects the DB unavoidability bundle generated in 07a to real
session runtime policy, hardens DB proxy listener authorization to the owning
agent session, and turns DB-generated bypass denials into dedicated audit
events.

This keeps the next increment useful without waiting for the full real
Postgres end-to-end harness. The real Postgres E2E remains the first item for
07c.

## Decisions

- Runtime bundle wiring is in 07b. A DB policy with unavoidability enabled
  should produce generated rules in the per-session policy engine.
- Listener authorization is based on session identity, not UID equality.
  `SO_PEERCRED` still supplies the peer PID and UID, but UID is recorded only
  for diagnostics and audit context.
- Unknown or mismatched listener peers fail closed when DB unavoidability is
  configured.
- DB bypass audit events are emitted only for denials whose rule metadata has
  `source: db_unavoidability`.
- Bypass event dedupe is an in-memory first pass with a 60 second window keyed
  by session, process identity, and destination tuple.
- 07b uses focused unit, API, and light proxy tests. Container-backed real
  Postgres validation is deferred to 07c.

## In Scope

- Generate the DB unavoidability bundle during session creation after the
  session ID is known.
- Merge generated rules and rule metadata into only the session-local policy
  engine.
- Wire Postgres proxy listener authorization through an injected session
  resolver.
- Add `peer_session_id` to listener auth failure lifecycle events when the
  resolver can identify the peer.
- Emit `db_bypass_attempt` lifecycle events for DB-generated bypass denials.
- Deduplicate repeated DB bypass events in memory for 60 seconds.
- Add focused tests for runtime policy merge, listener session auth, bypass
  event mapping, dedupe behavior, and Windows compilation.

## Out Of Scope

- A real Postgres server E2E test.
- Changing the default recommendation from observe to enforce.
- Supporting TCP DB listeners in enforce mode.
- Moving the DB proxy into a separate process.
- Persisting dedupe state across agent restarts.

## Runtime Bundle Wiring

Session creation currently compiles the policy engine before creating the
session. 07b changes that ordering for DB unavoidability policies:

1. Load and verify the policy file as today.
2. Decode the DB policy block with the existing DB policy decoder.
3. Create the session, obtaining its final `session_id`.
4. If DB unavoidability is disabled or there are no DB services, compile the
   loaded policy exactly as today.
5. If DB unavoidability is observe or enforce, build a `service.Config` from
   the decoded DB services using the same synthesized listener paths used by
   proxy startup.
6. Call `service.GenerateBundle` with the session ID and proxy identity.
7. Merge the generated policy rules and metadata into a copy of the loaded
   policy.
8. Compile the session-local policy engine with the existing policy variables.
9. Attach the compiled engine to the session.

The loaded policy object is not mutated globally. The merge happens on a
session-local policy value so later sessions do not inherit generated rules
with the wrong session ID.

If bundle generation or engine compilation fails after session creation, the
API destroys the partially-created session and returns the error. The session
must not remain visible with a missing or stale policy engine.

### Bundle Options

07b uses the generated session ID as `BundleOptions.SessionID`.

`BundleOptions.ProxySessionID` is set to a stable DB proxy identity used only
for generated rule ownership. The proxy still authorizes incoming clients by
the owning agent session ID. If a later plan moves the proxy out of process,
that identity can be replaced with a real proxy session ID without changing
the policy merge shape.

`BundleOptions.SocketBaseDir` is `filepath.Join(stateDir, "db-services")`.
Even though the current bundle generator uses each service listener path
directly, passing the base directory keeps the call aligned with the bundle
API.

`BundleOptions.IncludeToolRules` is true so generated bypass tool denials have
metadata that can drive `db_bypass_attempt` events.

`BundleOptions.Mode` comes from the DB unavoidability policy mode.

`BundleOptions.AllowHostnameOnlyInEnforce` remains false. Enforce mode should
not silently allow hostname-only upstreams without resolved destinations.

## Listener Session Authorization

The Postgres proxy gets an injected resolver:

```go
type SessionResolver interface {
	ResolveSessionID(pid int32) (sessionID string, ok bool)
}
```

The proxy config adds:

- `AgentSessionID string`
- `SessionResolver SessionResolver`

When DB unavoidability is observe or enforce, both values are required. Missing
resolver wiring is a startup/configuration error, not a fallback to UID checks.

For each accepted Unix socket connection:

1. Read peer credentials with `SO_PEERCRED`.
2. If peer credential read fails, close the connection and emit
   `db_listener_auth_fail` with reason `peercred_read_failed`.
3. Resolve the peer PID to a session ID.
4. If resolution fails, close the connection and emit
   `db_listener_auth_fail` with reason `session_unknown`.
5. If the resolved session ID does not match the configured agent session ID,
   close the connection and emit `db_listener_auth_fail` with reason
   `session_mismatch` and the resolved `peer_session_id`.
6. If it matches, continue to the normal proxy connection handler.

`peer_uid` and `peer_pid` remain on the lifecycle event for diagnostics.
`peer_session_id` is added when known.

This authorization applies in observe and enforce. Observe mode still exposes
the listener, so it must not accept traffic from unrelated sessions.

## Ptrace Session Resolver

The API layer wires the listener resolver from the ptrace tracer when ptrace is
available.

The tracer exposes:

```go
func (t *Tracer) ResolveSessionID(pid int32) (string, bool)
```

Resolution first checks an exact tracee key, then scans tracee state for a
matching process group/thread group ID. The method returns false when the PID
is not currently associated with a traced session.

If a process connects to the DB listener outside ptrace supervision, the
resolver returns false and the listener fails closed. That matches the DB
unavoidability threat model: only processes in the owning agent session should
use the listener.

Non-Linux builds keep compiling with no-op or absent resolver wiring behind
build tags. The Windows build remains a required verification command.

## DB Bypass Events

07b adds a `db_bypass_attempt` lifecycle event for denials tied to generated DB
unavoidability metadata.

The event includes:

- `session_id`
- `process_id`
- `process_identity`
- `db_service`
- `rule_name`
- `bypass_mode`
- `destination`
- `reason`
- `suppressed_count`

The event is emitted only when the policy decision is a deny and the decision
rule has metadata with:

```yaml
source: db_unavoidability
```

The metadata supplies the DB service name, bypass mode, and destination tuple.

### Event Paths

07b maps DB-generated denials where the existing enforcement path already has
the policy decision rule available:

- ptrace network denials
- transparent/network monitor denials
- generated bypass command denials

If a Unix socket or file path denial does not currently expose enough policy
decision metadata at the emission point, it remains a normal deny event in 07b.
The policy metadata still exists, and 07c can extend that path while adding the
real database E2E.

## Dedupe

The first implementation keeps an in-memory dedupe cache owned by the API or
event adapter layer.

The dedupe key is:

```text
session_id + process_identity + destination_tuple
```

Process identity uses the best available stable identity from the enforcement
path. Where only a PID is available in 07b, the identity is `pid:<pid>`.

The first event in a 60 second window is emitted immediately. Repeated matches
inside the window are suppressed and counted in memory. The next event emitted
after the window expires includes the suppressed count from the previous
window, then starts a new window.

The cache is bounded by time and opportunistic pruning on writes. It is not
persisted and does not need exact accounting across process restarts.

## Tests

Runtime bundle wiring tests:

- Session creation with DB unavoidability observe produces a policy engine
  containing generated DB metadata for the session ID.
- Session creation without DB unavoidability keeps existing behavior.
- Bundle generation failure after session creation destroys the partial
  session and returns an error.

Listener auth tests:

- Matching resolved session ID allows the connection handler path to continue.
- Resolver miss fails closed and emits `db_listener_auth_fail` with reason
  `session_unknown`.
- Resolver mismatch fails closed and emits `db_listener_auth_fail` with reason
  `session_mismatch` and `peer_session_id`.
- Peer credential read failure behavior remains covered.

Bypass event tests:

- A DB-generated deny emits `db_bypass_attempt` with DB service, bypass mode,
  destination, and rule name from metadata.
- A non-DB deny does not emit `db_bypass_attempt`.
- Duplicate DB bypass attempts inside 60 seconds are suppressed.
- The next event after the dedupe window carries the prior suppressed count.

Build verification:

- `go test ./internal/db/service ./internal/db/proxy/postgres ./internal/api -count=1`
- `GOOS=windows go build ./...`

## Success Criteria

- DB unavoidability policies generate session-local policy rules at runtime.
- The DB proxy listener rejects peers that are not resolved to the owning
  agent session.
- Auth failures include the peer session ID when known.
- DB-generated bypass denials create dedicated lifecycle audit events with
  dedupe.
- Existing behavior is unchanged for policies without DB unavoidability.
- Linux, macOS, and Windows builds still compile.
