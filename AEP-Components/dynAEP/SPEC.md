# dynAEP 1.0 Protocol Specification

**Document Version 1.0.0**
**Status: Authoritative**
**Supersedes: README.md sections 1-11 for normative reference**

This document is the authoritative protocol specification for dynAEP 1.0.
The reference implementation in this repository follows this specification.
Code that deviates from this document is incorrect. File issues for any
discrepancy found.

---

## 1. Overview

### 1.1 What dynAEP 1.0 Is

dynAEP 1.0 (Dynamic Agent Element Protocol) is a real-time event governance
protocol for multi-agent AI systems. It defines a deterministic bridge pipeline
that validates, orders and routes every event that enters or leaves an agent
system. All events pass through a pre-approved Action Lattice before reaching
any output renderer, agent backend or human-facing interface.

### 1.2 Purpose

The protocol solves three problems that arise when multiple autonomous agents
operate on shared state in real time.

Agent hallucination at the event level. Agents may propose actions that violate
system constraints, execute operations in the wrong order or reference events
that do not exist. The Action Lattice rejects such actions at arrival time.

Temporal disorder. Agents may produce timestamps that are stale, future-dated,
or inconsistent across distributed nodes. The bridge clock owns all time and
stamps every event authoritatively.

Causal confusion. Events from different agents may arrive in non-causal order
due to network latency or parallel execution. Vector clocks and a reorder
buffer restore the true causal order before any downstream processing.

### 1.3 Problems Solved

dynAEP 1.0 replaces ad-hoc event handling with a single deterministic pipeline.

Before dynAEP 1.0, event governance was fragmented across layers. UI mutations
were validated against the AEP scene graph. Temporal checks were applied later.
Agent actions had no structural validation at all. Each subsystem maintained its
own rules and events that passed one check could fail another with no clear
diagnostic path.

After dynAEP 1.0, every event from every source follows the same ordered
pipeline. Lattice validation runs first. Temporal authority runs second.
Structural validation runs third. Perception governance runs fourth. Events
that fail any stage are rejected immediately with a structured error. No
downstream system ever sees an ungoverned event.

### 1.4 Relationship to Other Protocols

dynAEP 1.0 is part of a family of open protocols for agentic systems.

AEP (Agent Element Protocol) provides build-time structural governance for
agent-created UIs. AEP defines the scene graph, registry and z-band topology
that dynAEP validates against at runtime. AEP governs structure at build time.
dynAEP governs events at runtime.

GAP (Governed Agentic Programming) defines instruction-level governance for
agent instructions. GAP specifies what an agent is told it may do. dynAEP
specifies what events an agent may actually produce. GAP governs the
instruction contract. dynAEP governs the execution contract.

AG-UI provides real-time event streaming between agent backends and frontend
renderers. dynAEP extends AG-UI events with validated lattice events, temporal
authority and perception governance.

MCP (Model Context Protocol) handles agent-to-tool connectivity and
discovery. dynAEP does not replace MCP. MCP manages tool invocation. dynAEP
governs the events those tools produce and consume.

## 2. Core Concepts

### 2.1 Action Lattice

The Action Lattice is a partial-order DAG (directed acyclic graph) of every
action the system can perform. It is the central governance mechanism in
dynAEP 1.0. Every event that enters the bridge is matched against the lattice
to determine its validity, its ordering constraints, its required trust level,
and its route to interested agents.

The lattice is defined declaratively in a YAML registry file
(registries/aep-lattice.yaml). It is loaded at bridge startup and may be
reloaded at runtime via the lattice governance configuration. The lattice is
read-only at runtime. Agents cannot add, remove or modify lattice nodes
dynamically. All changes must go through the registry file update process.

### 2.2 LatticeNode

A LatticeNode represents a single action in the lattice. Every node has these
properties.

path: A dot-delimited identifier that uniquely names the action. Paths use
colon as the delimiter for readability (e.g. webhook:incoming,
market:trade:execute). Paths are case-sensitive.

category: One of four categories that classify the action type.
 external_event - Events arriving from outside the system. These have no
 parents as they originate from external sources.
 system_event - Events generated by the system itself including startup
 sequences, health checks, validation steps and routing logic.
 agent_action - Actions proposed by agents. These require higher trust floors
 and more constraints than system events.
 output - Actions that produce human-facing output including notifications,
 UI mutations, speech and haptic feedback.

parents: A list of action paths that must be satisfied before this action can
fire. The lattice enforces a happens-before relationship. An action cannot fire
until all its parents have been satisfied. External events have empty parent
lists as they arrive from outside the system.

children: A list of action paths that may follow after this action. Children
define the forward traversal of the DAG. Not all children must fire. Children
represent possible continuations, not mandatory sequences.

constraints: A list of validation gates applied at event arrival time. Each
constraint has a type, a field, an optional condition and a description.
Supported constraint types are required_field, threshold, authorization, and
custom.

trust_floor: The minimum agent trust tier (1 through 5) required for an agent
to propose this action. The trust floor is checked before any other validation.

### 2.3 LatticeEvent

A LatticeEvent is the canonical event shape that flows through the bridge
pipeline. Every observer adapter normalises its native event into this shape.
Every agent action is encoded as a LatticeEvent before entering the pipeline.

Fields of a LatticeEvent:

source (string, required): Identifies the origin adapter or agent that produced
the event. Examples: "webhook:9000", "eth:0xabc...", "agent:trader-alpha".

action_path (string, required): Selects the LatticeNode that this event targets.
Must match a path declared in the lattice registry.

payload (object, required): Carries the event data. The structure of the
payload is action-specific. The bridge does not inspect payload structure
beyond constraint field checks.

bridge_timestamp (number, required): Set by the bridge at ingest time. This is
the authoritative time of arrival. Agents never set this field. The bridge
overwrites any value provided by the agent.

agent_id (string, optional): Identifies the originating agent if the event was
produced by an agent. Null for external events and system events.

trust_tier (number, optional): Conveys the agent current authorization level.
Set by the bridge based on the agent identity. Not settable by the agent.

### 2.4 LatticeFilterResult

The bridge produces a LatticeFilterResult for every LatticeEvent that passes
through the lattice filter. This is the structured verdict that downstream
pipeline stages use to determine routing and next actions.

Fields:

action_path (string): The path that was matched.
passed (boolean): True if the event passed all validation checks.
matched_node (string): The exact node path that was matched.
constraints_passed (array of string): Names of constraints that passed.
constraints_failed (array of string): Names of constraints that failed with
 reason strings.
partial_order_satisfied (boolean): True if all parent actions have occurred.
missing_parents (array of string): Paths of parent actions that have not been
 satisfied (empty if partial_order_satisfied is true).
trust_sufficient (boolean): True if the agent trust tier meets the node
 trust_floor.
matched_interests (array of AgentInterest): List of agent interests that match
 this event action path.
next_actions (array of string): Paths of child nodes that may follow.
duration_us (number): Wall-clock time spent in filter execution in microseconds.

### 2.5 AgentInterest

An AgentInterest is a registration record that declares an agent interest in
lattice paths matching a set of glob patterns. When an event action path
matches, the bridge routes the event according to the notification mode.

Fields:

agent_id (string, required): The agent unique identifier.
watch_paths (array of string, required): Glob patterns matching action paths.
 Supports exact match, single-segment wildcard (*) and multi-segment
 wildcard (**).
notify (string, required): One of "wake", "queue" or "log".
max_rate (string, optional): Rate limit string such as "10/min" or "1/sec".
constraints (object, optional): Additional match filters beyond the action
 path glob. Example: { "min_confidence": 0.7 }.

## 3. The Bridge Pipeline

The bridge pipeline is a strictly ordered sequence of validation and
enrichment stages. Every event follows every stage in order. There is no
short-circuit path and no stage skipping. The pipeline is deterministic:
given the same event and the same bridge state, the pipeline produces the
same result.

### 3.1 Stage 0: Template Instance Fast-Exit

Before any validation, the bridge checks whether the event targets a
pre-validated template instance. Template instances are AOT-validated elements
that have deterministic structure. Events targeting template instances skip
the entire runtime validation pipeline and proceed directly to output. This
is an optimization for high-frequency, low-risk mutations such as cursor
position updates and real-time coordinate streams.

The fast-exit check is O(1). It uses the event target_id to look up the
template registry. If a match is found and the template is still valid, the
event is stamped with bridge time and returned immediately. The fast-exit is
recorded in the evidence ledger for audit.

### 3.2 Stage 1: Lattice Filter

The lattice filter is the first and most important validation stage. It
validates every event against the Action Lattice before any other processing.
The filter runs seven checks in order.

1. Lattice membership. Does the event action_path match a declared lattice
 node? If not, the event is rejected immediately.

2. Trust floor check. Does the agent trust tier meet or exceed the node
 trust_floor? If not, the event is rejected.

3. Partial order check. Have all parent actions of this node been satisfied?
 The bridge maintains a satisfied set of action paths that grows as events
 pass through the filter. If any parent is missing from the satisfied set,
 the event is rejected with the list of missing dependencies.

4. Built-in constraint evaluation. Each constraint on the node is evaluated
 in order. For required_field constraints, the bridge checks that the named
 field exists in the event payload. For threshold constraints, the bridge
 evaluates the condition against the field value. For authorization
 constraints, the bridge checks the agent identity and trust tier against
 the condition.

5. Custom constraint evaluation. For constraints with type "custom", the
 bridge dispatches to the registered validation hook (see section 8).
 Custom constraints are evaluated last so that hooks receive the event
 after all built-in checks have passed.

6. Agent interest matching. The bridge matches the event action_path against
 all registered agent interests using glob matching. Matching interests are
 collected into the filter result.

7. Satisfied set update. If all checks pass, the node path is added to the
 satisfied set so that child actions can fire.

### 3.3 Stage 2: Temporal Authority

After lattice validation, the event passes through temporal authority. The
bridge clock stamps the event with authoritative time. Three checks run.

Drift check. The difference between agent-provided time (if any) and bridge
time must not exceed the configured max_drift_ms.

Future check. If the agent timestamp is more than max_future_ms ahead of
bridge time, the event is rejected.

Staleness check. If the event age exceeds max_staleness_ms, the event is
rejected.

The bridge timestamp is synthesized into the event as _temporal metadata.

### 3.4 Stage 3: Causal Ordering

Events that carry agent identity and sequence number pass through the causal
ordering engine. The engine assigns each agent a vector clock entry and
maintains a per-agent expected sequence number. Events with sequence numbers
matching expectations are delivered immediately. Events with sequence numbers
higher than expected are buffered in the reorder buffer. Events with sequence
numbers lower than expected are rejected as clock regression.

The reorder buffer has configurable capacity (default 64) and wait time
(default 200 ms). When the buffer fills, the oldest event by bridge time is
automatically delivered.

### 3.5 Stage 4: Structural Validation (AEP)

Events that target UI elements pass through AEP structural validation. The
bridge checks the element ID against the live scene graph, validates mutation
deltas against the element type schema and verifies z-band constraints. This
stage is unchanged from dynAEP v0.4.

### 3.6 Stage 5: Rego Policy Evaluation

The bridge evaluates a set of OPA/Rego policies against the event. Policies
are loaded from the configured policy path and evaluated via WASM or CLI
backend. The unified evaluator supports structural, temporal, perception, and
lattice policy files. All policy evaluations are cached with a configurable
decision cache.

For lattice policy evaluation, the bridge provides the following pre-computed
input fields to Rego so the policy does not duplicate the lattice registry:

- input.action_path: The lattice action path being evaluated.
- input.category: The action category (from the lattice node).
- input.trust_tier: Agent trust tier (1-5).
- input.parents_of: Array of parent paths for this action (computed by the
 bridge from the lattice DAG at filter time).
- input.is_root: Boolean (true if this action has no parents).
- input.all_actions: All known action paths in the lattice.
- input.satisfied_actions: Array of action paths that have been validated and
 marked as satisfied (maintained by the bridge's per-agent satisfied set).

This bridge-supplied contract replaces the hardcoded parent_of() and
root_action() helpers that existed in the lattice-policy.rego draft. The
Rego policy consumes bridge-computed data rather than duplicating the
aep-lattice.yaml registry. This eliminates drift between the YAML
registry and the Rego policy.

### 3.7 Stage 6: Content Scanning

Events that carry text payload pass through the content scanner. The unified
scanner uses Aho-Corasick pattern matching against configured pattern sets
including secrets detection, injection patterns and PII patterns. Findings
with severity "hard" cause immediate rejection.

### 3.8 Stage 7: Perception Governance (dynAEP-TA-P)

Events that produce human-facing output pass through perception governance.
The perception engine validates temporal properties against published
perceptual bounds. Speech pacing, haptic duration, notification cadence, and
audio composition are validated against the perception registry. Adaptive
user profiles adjust bounds based on learned user preferences.

### 3.9 Stage 8: Evidence Recording

All accepted events, rejected events and fast-exit events are recorded to the
evidence ledger. The ledger is a buffered, hash-chained record of every event
that entered the bridge. The ledger supports audit queries and operational
analytics.

## 4. Action Lattice Protocol

### 4.1 Lattice Structure

The Action Lattice is a directed acyclic graph. Every node is an action.
Edges are parent-child relationships. The lattice must be acyclic. Cycles are
detected at startup and cause the bridge to refuse to load.

The lattice is defined in registries/aep-lattice.yaml. The registry file has
a top-level actions map. Each key is the action path. Each value is the node
definition. The file also carries metadata fields:

aep_version (string): Must be "1.2" or later.
dynaep_version (string): Must be "1.0" or later.
lattice_revision (integer): Monotonically increasing revision number. The
 bridge logs a warning if the revision decreases across reloads.

### 4.2 Action Categories

Every action belongs to exactly one category. The category defines the basic
behavior of the action within the pipeline.

external_event: Events that originate outside the system. These are the root
nodes of the lattice. They have no parents. External events are always
accepted at the lattice level (subject to constraint evaluation). Examples:
webhook:incoming, blockchain:event, email:incoming, sensor:reading,
market:price:update.

system_event: Events generated by the system itself. These include startup
sequences, health checks, validation steps, routing logic and scheduled
tasks. System events may have parents and children. Examples:
system:startup, system:health:check, system:ready, action:validate,
action:route.

agent_action: Actions proposed by agents. These require higher trust floors
and more constraints. Agent actions form the most complex chains in the
lattice. Examples: agent:register, agent:propose_action,
agent:interest:register, agent:trade:propose, agent:email:classify.

output: Actions that produce human-facing output. These are leaf nodes or
near-leaf nodes. Output actions are validated against the AEP scene graph
(for UI) or the perception registry (for speech, haptic, notification).
Examples: output:notify, output:ui_mutation, output:speech, output:haptic.

### 4.3 Partial-Order DAG

The lattice enforces a happens-before relationship across all actions. An
action cannot fire until ALL its parent actions have been satisfied. The
bridge tracks satisfied actions in a set that grows as events pass through
the filter.

When an event arrives, the bridge performs these partial-order checks.

1. Lower set computation. The bridge computes all ancestors of the event
 action path by following parent references backward through the DAG.

2. Parent verification. Every direct parent of the event node is checked
 against the satisfied set. If any direct parent is missing, the event is
 rejected with a list of missing parents.

3. Indirect ancestor verification is not required. Only direct parents must
 be satisfied. This prevents deep dependency chains from blocking actions
 when intermediate ancestors were satisfied indirectly.

Upper set traversal (forward closure) is used for routing. When the bridge
computes which agents should receive an event, it follows children forward
from the current node to determine what actions may follow and which agents
are interested in those downstream actions.

### 4.4 Constraint Types

Every constraint in a lattice node has a type. Four types are defined.

required_field: The named field must exist in the event payload. The field
value may be any type including null. The bridge checks for key existence
only. Example: field "signature" must exist for webhook:incoming.

threshold: The named field value must satisfy the condition expression.
Conditions are string expressions evaluated against the field value. Supported
operators: >, >=, <, <=, ==, between, within_range, defined, exists, and
custom predicates registered via the validation hook interface.

authorization: The named field identity or trust tier must satisfy the
condition. Authorization constraints are evaluated against the bridge agent
registry and the agent trust tier. Conditions may reference trust_tier,
agent_type or custom capability checks.

custom: Dispatched to the registered validation hook. The hook receives
the full event, the full lattice and the matched node. The hook returns a
HookResult. Custom constraints are evaluated after all built-in constraints
for the node.

### 4.5 Trust Tiers

Trust tiers are integers from 1 to 5. Higher numbers indicate greater trust.
The trust floor of a lattice node is the minimum tier required to propose
that action.

Tier 1. Any agent or external system. Root-level external events including
webhooks, blockchain events, email receipts, sensor readings and system
startup events require tier 1. All external observers produce events at tier 1.

Tier 2. Registered agents with verified capabilities. Agent registration,
system readiness signals, UI mutations, speech output, email classification,
and sensor analysis require tier 2.

Tier 3. Trusted agents with demonstrated reliability. Agent action proposals,
email drafting and market price analysis require tier 3.

Tier 4. High-trust agents with authorization. Email review, email sending,
system shutdown and market price analysis require tier 4.

Tier 5. Maximum trust. Trade proposals, trade validation, trade execution,
and any action that moves value or changes system authority require tier 5.

The trust tier is checked at event arrival time before any other validation.
If the agent trust tier is below the action trust_floor, the event is rejected
and a LATTICE_FILTER_RESULT with trust_sufficient=false is returned.

### 4.6 Lattice Loading and Reloading

The lattice is loaded from the registry file at bridge startup. The bridge
performs validation on load including cycle detection, orphan detection
(parents that do not exist as nodes), constraint syntax validation and trust
floor range checks (must be 1-5).

The lattice may be reloaded at runtime via the governance configuration
interface. During reload, the bridge computes the diff between the old and
new lattice. Only additive changes (new nodes added, new children added to
existing nodes) are applied without restart. Destructive changes (node
removal, parent modification, trust floor reduction) require a bridge restart
and are logged as fatal configuration errors.

### 4.7 Satisfied Set Persistence

The satisfied set is the set of action paths that have been satisfied by
events that passed the lattice filter. The satisfied set is ephemeral. It
is reset on bridge restart.

For persistent tracking across restarts, external systems should replay their
action history through the bridge. The bridge recovery protocol (section 6.6)
supports replay of the reorder buffer but does not persist the satisfied set.

## 5. Event Protocol

### 5.1 Event Namespace

All dynAEP 1.0 events use the dynaep_type field to identify their protocol
type. The type field follows the AG-UI convention ("STATE_DELTA", "CUSTOM",
"QUERY"). The dynaep_type field within CUSTOM events carries the specific
protocol type.

### 5.2 LATTICE_EVENT

The canonical event shape. Every observer adapter normalises its native event
into this shape before forwarding to the bridge.

JSON Schema:

{
 "type": "CUSTOM",
 "dynaep_type": "LATTICE_EVENT",
 "source": "<adapter or agent name>",
 "action_path": "<dot-delimited lattice path>",
 "payload": { <event-specific data> },
 "bridge_timestamp": <epoch ms>,
 "agent_id": "<agent id or null>",
 "trust_tier": <1-5>
}

The source field identifies the origin. The action_path field selects the
lattice node. The payload field carries event-specific data. The
bridge_timestamp field is set by the bridge at ingest time and overwrites
any agent-provided value. The agent_id field is null for external and system
events. The trust_tier field is set by the bridge from the agent registry.

### 5.3 LATTICE_FILTER_RESULT

The structured verdict produced by the lattice filter for every event.

JSON Schema:

{
 "type": "CUSTOM",
 "dynaep_type": "LATTICE_FILTER_RESULT",
 "action_path": "<matched path>",
 "passed": <true|false>,
 "matched_node": "<exact node path>",
 "constraints_passed": ["<constraint name>"],
 "constraints_failed": ["<reason string>"],
 "partial_order_satisfied": <true|false>,
 "missing_parents": ["<parent path>"],
 "trust_sufficient": <true|false>,
 "matched_interests": [
 {
 "agent_id": "<agent id>",
 "watch_paths": ["<glob pattern>"],
 "notify": "wake|queue|log"
 }
 ],
 "next_actions": ["<child path>"],
 "duration_us": <microseconds>
}

When passed is false, the event is rejected immediately. The constraints_failed
array contains human-readable reason strings. When partial_order_satisfied is
false, missing_parents lists the paths that must be satisfied first.

### 5.4 LATTICE_REGISTER

Used by agents to declare interest in lattice paths.

JSON Schema:

{
 "type": "CUSTOM",
 "dynaep_type": "LATTICE_REGISTER",
 "agent_id": "<agent id>",
 "watch_paths": ["<glob pattern>"],
 "notify": "wake|queue|log",
 "max_rate": "<rate limit string>",
 "constraints": { <additional filters> }
}

The notify field selects the notification mode. wake immediately triggers the
agent with the event payload. queue stores the event in a per-agent queue for
later retrieval. log records the event in the agent activity log without
delivery.

The max_rate field sets a rate limit in the format "<number>/<unit>" where
unit is sec, min, hour. Events exceeding the rate limit are queued or dropped
depending on the notify mode.

LATTICE_REGISTER events themselves pass through the lattice filter. They must
match the agent:interest:register action path which requires agent:ready as a
parent. This ensures that only properly authenticated and initialized agents
can subscribe to event streams.

### 5.5 DYNAEP_REJECTION

Emitted when the bridge rejects an event at any pipeline stage.

JSON Schema:

{
 "type": "CUSTOM",
 "dynaep_type": "DYNAEP_REJECTION",
 "target_id": "<event target id>",
 "error": "<human-readable reason>",
 "original_event_timestamp": <epoch ms>,
 "stage": "lattice|temporal|causal|structural|rego|scanner|perception",
 "violations": ["<detailed violation string>"]
}

The stage field identifies which pipeline stage rejected the event. The
violations array carries individual violation details for multi-check stages.

### 5.6 AEP Event Families (Unchanged from v0.4)

All existing event types from dynAEP v0.4 remain fully supported. They flow
through the same pipeline with lattice validation as the first stage.

AEP_MUTATE_STRUCTURE: Structure mutations against the AEP scene graph.
Carries element_id, mutation type and mutation delta.

AEP_MUTATE_BEHAVIOUR: Behaviour mutations. Carries element_id, behaviour
name and behaviour configuration.

AEP_MUTATE_SKIN: Skin mutations. Carries element_id, skin property, and
new value.

AEP_QUERY: Runtime scene graph queries. Carries query type and parameters.

AEP_QUERY_RESULT: Query responses. Carries the query result data.

AEP_RUNTIME_COORDINATES: Rendered coordinates from ResizeObserver and
MutationObserver. Carries element_id and coordinate data.

### 5.7 Temporal Authority Event Families (Unchanged from v0.4)

AEP_CLOCK_SYNC: Clock synchronization notification. Carries sync source,
offset and latency.

AEP_TEMPORAL_STAMP: Temporal stamp annotation. Attached to every accepted
event by the temporal validator.

DYNAEP_TEMPORAL_REJECTION: Temporal validation failure. Carries violation
type and threshold details.

DYNAEP_CAUSAL_VIOLATION: Causal ordering violation. Carries violation type,
agent_id, expected and received sequence numbers.

AEP_TEMPORAL_FORECAST: TimesFM forecast result. Carries element_id,
forecast values and confidence intervals.

AEP_TEMPORAL_ANOMALY: Anomaly detection result. Carries element_id, anomaly
score and recommendation.

AEP_TEMPORAL_RESET: Temporal state reset. Carries reset scope and reason.

AEP_TEMPORAL_RECOVERY: Recovery phase event. Carries phase number and status.

AEP_AGENT_REREGISTER: Agent re-registration during recovery. Carries agent_id
and sequence counter.

AEP_REREGISTER_RESULT: Re-registration result. Carries agent_id and status.

### 5.8 Perception Governance Event Families (Unchanged from v0.4)

AEP_PERCEPTION_GOVERNED: Governed envelope produced by the perception engine.
Carries modality, original parameters and governed parameters.

DYNAEP_PERCEPTION_REJECTION: Perception validation failure. Carries
modality, violation type and threshold details.

AEP_PERCEPTION_PROFILE_UPDATE: Adaptive profile update. Carries agent_id,
modality and adjusted parameters.

## 6. Temporal Authority Protocol

### 6.1 Bridge Clock

The BridgeClock is the sole authoritative time source for the entire protocol
stack. Agents never generate timestamps. All timestamps are set by the bridge
at ingest time.

The clock synchronizes to an external reference with automatic fallback.

PTP (IEEE 1588). Provides microsecond precision for mission-critical
industrial deployments. The clock reads the PTP offset from sysfs.

NTP (default). Provides millisecond precision via SNTP. Synchronization runs
in a background daemon thread and never blocks the main event processing
thread. The NTP source defaults to pool.ntp.org but is configurable.

System clock. Fallback when no network time source is available. The clock
logs a warning on fallback activation.

Clock corrections are applied via slewing (gradual adjustment) for corrections
between 1 ms and 1000 ms. Corrections below 1 ms are ignored. Corrections
above 1000 ms are applied as immediate steps with a logged warning.

### 6.2 BridgeTimestamp

Every event that enters the bridge receives a BridgeTimestamp.

Fields:

bridge_time_ms (number): The authoritative time in milliseconds since Unix
epoch. Always set by the bridge clock.

agent_time_ms (number or null): The original timestamp provided by the agent.
Null if the agent did not provide a timestamp.

drift_ms (number): The difference between agent time and bridge time when
both are available. Zero when agent time is null.

source (string): The protocol that produced the timestamp: "ntp", "ptp",
or "system".

synced_at (number): The Unix timestamp of the last successful clock sync.

### 6.3 Timestamp Validation

Every incoming event passes through temporal validation. The validation
pipeline runs these steps in order.

1. Read the agent-provided timestamp from the event timestamp field. The field
 may be absent.

2. Produce the authoritative BridgeTimestamp from the bridge clock.

3. Overwrite the event timestamp with bridge time when bridge_is_authority is
 true (the default).

4. Drift check. If the agent provided a timestamp and the absolute difference
 between agent time and bridge time exceeds max_drift_ms (default 50 ms),
 a drift violation is recorded.

5. Future check. If the agent timestamp exceeds bridge time by more than
 max_future_ms (default 500 ms), a future timestamp violation is recorded.

6. Staleness check. If the bridge time exceeds the agent timestamp by more
 than max_staleness_ms (default 5000 ms), a stale event violation is
 recorded.

Three enforcement modes control how violations are handled.

strict: Any violation causes the event to be rejected.
permissive: Violations are converted to warnings. The event is accepted.
log_only: All events are accepted. Violations are logged silently.

### 6.4 TIM Clock Quality Tracking

The ClockQualityTracker provides IETF TIM-compatible clock quality metadata.
It implements the TIM sync state machine with three states.

LOCKED: The clock is synchronized and tracking within tolerance.
HOLDOVER: The clock has lost sync but is maintaining the last known offset
 with increasing uncertainty.
FREEWHEEL: The clock has lost sync for an extended period. Timestamps carry
 maximum uncertainty.

Confidence classes A through F indicate the reliability of the current time:

A: PTP with sub-microsecond uncertainty.
B: PTP with sub-millisecond uncertainty.
C: NTP with sub-10-ms uncertainty.
D: NTP with sub-100-ms uncertainty.
E: System clock or high-uncertainty NTP or PTP.
F: FREEWHEEL state. Timestamps are unreliable.

Anomaly flags detect unusual clock behavior:

LARGE_STEP: A single sync correction exceeded 1000 ms.
HIGH_JITTER: A sync correction exceeded 3 standard deviations from the mean.
SOURCE_CHANGE: The sync source changed (e.g. PTP to NTP fallback).
SYNC_LOSS: The clock entered FREEWHEEL state.

Welford online algorithm computes streaming variance without storing the
full measurement history.

### 6.5 Workflow Temporal Primitives

Four primitives use bridge-authoritative time instead of system clocks. All
primitives accept a getNow function for dependency injection, making them
fully testable with mock clocks.

TemporalDeadline: A bridge-time deadline. The caller sets a deadline by bridge
time and is notified when it elapses. Used for task timeouts and SLA
enforcement.

TemporalSchedule: A recurring schedule based on bridge time. The schedule
fires at bridge-authoritative intervals. Used for periodic tasks and health
checks.

TemporalSleepResume: A sleep operation that uses bridge time. The sleep
duration is measured in bridge milliseconds. Used for controlled delays in
workflow execution.

TemporalTimeout: A timeout wrapper for async operations. The timeout is
measured in bridge milliseconds. Used to detect hung operations.

### 6.6 Bridge Recovery Protocol

Three-phase recovery for graceful bridge restarts.

Phase 1 (Announce). The bridge emits a AEP_TEMPORAL_RECOVERY event with
phase=1. All agents detect the recovery and prepare to re-register their
sequence counters.

Phase 2 (Re-register). Each agent emits an AEP_AGENT_REREGISTER event with
its agent_id and last sequence counter. The bridge replays the persisted
reorder buffer to bring the causal ordering state up to date.

Phase 3 (Resume). The bridge emits a AEP_TEMPORAL_RECOVERY event with
phase=3. Normal event processing resumes. Events that arrived during
recovery are validated against the re-established state.

The recovery protocol detects the storage backend automatically (file,
SQLite, external KV). All backends support the same three-phase recovery
contract.

## 7. Causal Ordering Protocol

### 7.1 Vector Clocks

Every agent registered with the bridge receives a vector clock entry. The
vector clock maps agent IDs to sequence numbers. Each event from an agent
carries the agent vector clock snapshot at the time the event was produced.

The bridge uses sparse vector clock representation. Only agents with non-zero
sequence numbers are tracked. This provides O(min(|A|, |B|)) comparison
instead of O(totalAgents).

The vector clock supports four operations.

increment(agent_id): Advance the calling agent sequence number by 1.

merge(other): Absorb all entries from another vector clock. For each entry,
take the maximum of the two values.

dominates(other): Return true if this clock is strictly ahead of the other
clock on all entries and ahead on at least one entry.

is_concurrent_with(other): Return true if neither clock dominates the other.
Used for conflict detection.

### 7.2 Reorder Buffer

The reorder buffer holds out-of-order events while waiting for their
predecessors to arrive. The buffer has configurable capacity (default 64)
and maximum wait time (default 200 ms).

When an event arrives with a sequence number higher than expected, the event
is placed in the reorder buffer. The bridge waits for the missing predecessor
up to max_reorder_wait_ms. If the predecessor arrives in time, the buffered
event is delivered in order.

When the buffer reaches capacity, the oldest event by bridge timestamp is
flushed and delivered regardless of ordering. This prevents the buffer from
blocking the pipeline when a predecessor is permanently lost.

### 7.3 Conflict Resolution

When two agents produce concurrent mutations on the same element (detected
via vector clock concurrency), the bridge applies the configured conflict
resolution strategy.

last_write_wins (default): The event with the later bridge timestamp is
accepted. The earlier event is rejected with a DYNAEP_CAUSAL_VIOLATION.

optimistic_locking: The first event to arrive is accepted. The second event
is rejected. The bridge provides the winning event metadata so the losing
agent can retry with updated state.

### 7.4 Durable Causal State

All causal ordering state persists across bridge restarts. Three storage
backends are supported.

FileBasedCausalStore: JSONL append log with periodic compaction. Default
backend. Suitable for single-node deployments.

SqliteCausalStore: SQLite with WAL mode and transactions. Suitable for
deployments needing atomicity guarantees.

ExternalCausalStore: Adapter for external key-value stores including Redis,
DynamoDB and Cloudflare KV. Suitable for multi-node deployments.

## 8. Validation Hook Protocol

### 8.1 Hook Interface

Every validation hook implements the ValidationHook interface.

```
interface ValidationHook {
 name: string;
 version: string;

 validate(
 event: LatticeEvent,
 lattice: ActionLattice,
 node: LatticeNode,
 ): Promise<HookResult>;
}
```

The validate method receives three arguments.

event: The incoming LatticeEvent. The hook may inspect any field including
the action_path, payload, source and agent_id.

lattice: The full ActionLattice. The hook may traverse the lattice to access
ancestor and descendant nodes, constraints and trust floors.

node: The specific LatticeNode matched by the event action_path. The hook
may inspect node constraints and properties.

### 8.2 HookResult

The validate method returns a HookResult.

```
interface HookResult {
 passed: boolean;
 score: number; // 0.0 (invalid) to 1.0 (perfect)
 confidence: number; // 0.0 (uncertain) to 1.0 (fully confident)
 adjustments?: Record<string, unknown>;
 details?: string;
}
```

passed: Boolean pass or fail. A hook that returns passed=false causes the
event to be rejected.

score: A quality score from 0.0 to 1.0. Scores below 0.5 are treated as
failures in consensus mode.

confidence: The hook confidence in its own evaluation from 0.0 to 1.0.
Low confidence indicates the hook lacks sufficient data to evaluate.

adjustments: Optional payload modifications applied to the event on pass.
Hooks use adjustments to enrich the event with additional metadata.

details: Human-readable explanation of the hook verdict.

### 8.3 Hook Registry

The HookRegistry manages registered hooks. Operations:

register(hook: ValidationHook): Register a hook. Registration fails if a
hook with the same name already exists.

get(name: string): ValidationHook | null: Retrieve a registered hook by name.

unregister(name: string): void: Remove a hook from the registry.

list(): ValidationHook[]: List all registered hooks.

clear(): void: Remove all hooks.

### 8.4 Dispatch Rule

When the lattice filter encounters a constraint of type "custom", it looks
up the named hook in the HookRegistry. The constraint metadata specifies
the hook name in a hook_name field. If the hook is not found, the bridge
emits a warning and treats the constraint as failed.

The bridge supports three dispatch modes.

fail-fast (default): Stop on the first hook that returns passed=false.
The event is rejected immediately with the failing hook details.

consensus: Run all hooks regardless of individual results. Collect all
HookResult objects. The event passes if at least one hook returns passed=true
with score >= 0.5.

log-only: Run all hooks. All results are logged. The event always passes
the custom constraint stage regardless of hook verdicts.

### 8.5 MLE Validation Hook (Reference Implementation)

The MLE (Machine Learning Evaluation) hook is the reference validation hook.
It uses attractor indexing to evaluate events against learned patterns. The
hook computes a similarity score between the event feature vector and stored
attractors from the Lattice Memory. Events scoring below the similarity
threshold are flagged for review.

The MLE hook registers as "mle-validator" version "1.0.0". It is executed
for every custom constraint in the lattice when configured.

### 8.6 Hook Best Practices

Hooks must be deterministic. Given the same event, lattice and node, a hook
must produce the same HookResult. Non-deterministic hooks break replay and
audit.

Hooks must be fast. The target is sub-millisecond per event. Hooks that
perform I/O should cache results or use batch processing.

Hooks must never throw. All errors must be caught and returned as a
HookResult with passed=false and a descriptive details string.

Hooks must be versioned. Use semver. The bridge logs hook versions on
startup and includes them in audit trails.

## 9. Observer Adapter Protocol

### 9.1 Adapter Interface

Every observer adapter implements the ObserverAdapter interface.

```
interface ObserverAdapter {
 name: string;
 start(): Promise<void>;
 stop(): Promise<void>;
 onEvent(callback: (event: LatticeEvent) => void): void;
}
```

name: A unique identifier for this adapter instance. Used as the source
field in LatticeEvent objects.

start(): Begin consuming events from the external source. Must be idempotent.
Calling start() on an already-started adapter is a no-op.

stop(): Release all resources including server sockets, timers, and
connections. Must be idempotent and safe to call even if not started.

onEvent(callback): Register the callback that receives normalized LatticeEvent
objects. Only one callback is supported. Subsequent calls replace the
previous callback.

### 9.2 Adapter Lifecycle

The adapter lifecycle has four phases.

1. Construction. The adapter is instantiated with its configuration.
 Configuration validation happens in the constructor. Fatal configuration
 errors throw at construction time so they are caught at setup time.

2. Registration. The bridge calls onEvent on the adapter to register the
 event callback. The adapter stores the callback for later invocation.

3. Running. The bridge calls start(). The adapter begins consuming from the
 external source. Each incoming event is normalized into a LatticeEvent and
 forwarded to the registered callback.

4. Shutdown. The bridge calls stop(). The adapter tears down all resources.
 After stop, no further events are forwarded.

### 9.3 LatticeEvent Normalization

Every adapter must normalize its native event format into the canonical
LatticeEvent shape. The normalization contract:

source: Set to the adapter name. Must be stable across restarts.

action_path: Set to the lattice path that corresponds to this event. The
adapter determines the path from the event type, endpoint path or a
configured mapping.

payload: Set to the event-specific data. The adapter may transform the
payload for consistency (e.g. field name normalization).

bridge_timestamp: Set to Date.now() at normalization time. The bridge
overwrites this with its own clock during temporal validation, but the
adapter sets an initial value for the interim.

agent_id: Set to null. The bridge determines the agent identity.

### 9.4 Built-in Adapters

Five adapters ship with the reference implementation.

WebhookAdapter: Receives HTTP POST requests at a configurable endpoint.
Validates cryptographic signatures (HMAC, RSA, Ed25519). Routes the request
body into a LatticeEvent with action path determined by the endpoint path
or a request header. Supports rate limiting per source IP.

SSEAdapter: Connects to external Server-Sent Event streams. Parses event
types and data fields into LatticeEvents. Supports reconnection with
exponential backoff. Useful for market data feeds, social media streams,
and real-time sensor networks.

PollAdapter: Polls external APIs at configurable intervals. Uses an
idempotency key to deduplicate consecutive polls. Supports HTTP, database
queries and file system watches. Configurable poll interval with jitter
to avoid thundering herd.

BlockchainAdapter: Connects to blockchain nodes (Ethereum, Solana) via
JSON-RPC or WebSocket. Monitors logs, transactions and contract events.
Verifies transaction hashes and block confirmations before emitting events.
Supports multiple chain configurations.

EmailAdapter: Connects to IMAP or SMTP servers. Monitors inbox folders for
new messages. Parses MIME structure into structured payloads. Supports TLS
and OAuth2 authentication. Configurable polling or IDLE mode.

### 9.5 Custom Adapter Development

To create a custom adapter:

1. Create a new directory under observers/ with an index file implementing
 ObserverAdapter.

2. Define a configuration interface with connection parameters.

3. Implement start() to establish the connection and begin consuming.

4. Implement stop() to tear down all resources.

5. Call the registered onEvent callback for each normalized LatticeEvent.

6. Handle errors gracefully. Log failures but do not throw from event
 processing paths. Implement retry logic for transient failures.

## 10. Agent Interest Protocol

### 10.1 Registration Schema

Agents declare interest in lattice paths via the LATTICE_REGISTER event.
The registration carries these fields.

agent_id (string, required): The agent unique identifier. Must match a
registered agent.

watch_paths (array of string, required): Glob patterns matching action paths.
The bridge evaluates these patterns against every event action_path.

notify (string, required): One of three notification modes.

 wake: The agent is immediately invoked with the event payload. Used for
 real-time responses.

 queue: The event is stored in a per-agent queue. The agent polls the queue
 on its own schedule.

 log: The event is recorded in the agent activity log without delivery.
 Used for audit trails and analytics.

max_rate (string, optional): Rate limit in the format "<number>/<unit>".
Examples: "10/min", "1/sec". Events exceeding the rate limit are queued
(wake mode) or dropped (log mode).

constraints (object, optional): Additional match filters beyond the action
path glob. Example: { "min_confidence": 0.7, "max_amount": 10000 }.

### 10.2 Glob Matching

The bridge supports three glob pattern levels.

Exact match: "email:incoming" matches only that exact path.

Single-segment wildcard: "market:*" matches any single-level path under
market. "market:price:update" does NOT match because it has two segments
after market. "market:price" matches.

Multi-segment wildcard: "email:**" matches any path under email at any
depth. "email:incoming", "email:classify" and "email:draft_reply" all match.

### 10.3 Registration Flow

Agents register their interests by sending a LATTICE_REGISTER event. This
event passes through the lattice filter and must match the
agent:interest:register action path.

The agent:interest:register node has agent:ready as its parent. This means
the agent must have already been registered with the system (via agent:register
and agent:ready) before it can subscribe to event streams. This ensures that
only properly authenticated and initialized agents can declare interests.

### 10.4 Interest Persistence

Agent interests persist across bridge restarts when persistence is enabled
in the configuration. The persistence file stores all registrations and
restores them on startup. The bridge replays all registered interests before
accepting any events.

## 11. Configuration Protocol

### 11.1 Configuration File

All bridge behaviour is governed by the dynaep-config.yaml file. This file
uses YAML format and is loaded at bridge startup. The configuration defines
transport, lattice, validation, temporal, perception and all other bridge
subsystems.

### 11.2 Reference to CONFIG.md

The complete configuration schema including all fields, defaults and valid
values is documented in CONFIG.md at the repository root. This specification
does not inline configuration details. CONFIG.md is the authoritative
configuration reference.

See CONFIG.md for:

- Full configuration file schema
- Default values for all settings
- Valid value ranges and enums
- Environment variable overrides
- Configuration reload behavior

### 11.3 Configuration Loading Order

The bridge loads configuration in this order with later values overriding
earlier ones.

1. Built-in defaults compiled into the bridge binary.
2. Configuration file at the configured path (default: ./dynaep-config.yaml).
3. Environment variable overrides (prefixed with DYNAEP_).
4. Command-line arguments.

### 11.4 Runtime Configuration Reload

The bridge supports runtime reload of most configuration sections. Sections
that require restart are marked in CONFIG.md. The reload endpoint is
available via the bridge HTTP management interface.

On reload, the bridge validates the new configuration against the current
system state. Invalid configurations are rejected with an error response
and the running configuration is unchanged.

## 12. Security Considerations

### 12.1 Trust Tier Enforcement

The trust tier is the primary security mechanism in dynAEP 1.0. Every action
in the lattice has a minimum trust tier. Agents cannot propose actions at a
tier above their assigned level.

Trust tiers are assigned during agent registration. The bridge validates the
agent identity against the agent registry. Trust tiers cannot be self-assigned.
The bridge manages all trust tier assignments.

### 12.2 Signature Validation

Webhook adapters validate cryptographic signatures on incoming requests.
Three signature schemes are supported.

HMAC-SHA256: Shared-secret signing. The adapter compares the computed HMAC
against the signature header.

RSA-SHA256: Public-key signing. The adapter verifies the signature using the
configured public key.

Ed25519: EdDSA signing. The adapter verifies the signature using the
configured public key.

Events with invalid or missing signatures are rejected before lattice
validation.

### 12.3 Authorization Constraints

Lattice nodes may specify authorization constraints that check agent
credentials against the action requirements. Authorization constraints are
evaluated during lattice filter execution and may reference:

- Agent trust tier
- Agent type (builder, reviewer, trader, etc.)
- Agent capabilities declared during registration
- Custom authorization hooks

### 12.4 Rate Limiting

The bridge enforces rate limits at two levels.

Per-source rate limiting: The webhook adapter rate-limits by source IP.
Configuration defines the maximum requests per second per IP.

Per-agent rate limiting: Agent interests specify a max_rate. Events matching
an interest that exceeds the rate limit are queued or dropped.

### 12.5 Content Scanning

All event payloads containing text pass through the content scanner. The
scanner uses Aho-Corasick pattern matching against configured pattern sets
including:

Secrets detection: API keys, tokens, passwords, private keys and other
sensitive credentials. Patterns are regex-based and configurable.

Injection detection: SQL injection, command injection, template injection,
and cross-site scripting patterns.

PII detection: Email addresses, phone numbers, credit card numbers, social
security numbers and other personally identifiable information patterns.

Events with hard-severity findings are rejected. Soft-severity findings are
attached as metadata to the event.

### 12.6 Evidence Ledger

Every accepted and rejected event is recorded in the evidence ledger. The
ledger is hash-chained: each entry carries the hash of the previous entry.
This provides tamper-evident audit trails for all bridge operations.

The ledger supports:
- Append-only writes
- Cryptographic chain verification
- Configurable buffer and flush intervals
- Multiple storage backends

### 12.7 Clock Security

The bridge clock is the sole authoritative time source. Agents cannot set
timestamps. The clock synchronizes to external references with automatic
fallback. Clock corrections are logged and monitored for anomalies.

The TIM clock quality tracker provides confidence classes for every
timestamp. Downstream systems can use the confidence class to determine
whether to trust a timestamp for time-critical operations.

### 12.8 Replay Protection

Causal ordering prevents event replay attacks. Each agent has a monotonically
increasing sequence number. The bridge rejects events with sequence numbers
at or below the expected value. This prevents an attacker from replaying a
captured event with a sequence number that has already been delivered.

The sequence number check is enforced independently of the vector clock.
Both must pass for an event to be accepted through causal ordering.

## Appendix A: Glossary

Action Lattice: A partial-order DAG of all system actions. The central
governance mechanism in dynAEP 1.0.

Agent: An autonomous software entity that can propose actions, register
interests and consume events.

Bridge: The deterministic pipeline that validates, orders and routes every
event.

Bridge Clock: The sole authoritative time source for the entire protocol
stack.

Causal Ordering: The enforcement of happens-before relationships across
multi-agent event streams using vector clocks.

LatticeEvent: The canonical event shape that flows through the bridge
pipeline.

LatticeFilterResult: The structured verdict from the lattice filter.

LatticeNode: A single action in the Action Lattice with path, category,
parents, children, constraints and trust floor.

ObserverAdapter: A pluggable component that normalises external event sources
into LatticeEvents.

Partial Order: The happens-before relationship between lattice nodes. An
action cannot fire until all its parent actions have been satisfied.

Satisfied Set: The set of action paths that have been satisfied by events
that passed the lattice filter.

Tim: Temporal Integrity Metadata. IETF-compatible clock quality metadata.

Trust Tier: An integer from 1 to 5 that governs which agents can propose
which actions.

ValidationHook: A pluggable module that inspects a LatticeEvent against a
LatticeNode and produces a pass/fail verdict.

Vector Clock: A per-agent sequence number map used to determine causal
ordering and detect concurrent events.
